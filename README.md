# Kaiten Installer

macOS-установщик сервиса автоматизации работы с Kaiten и Пачкой.

## Что делает сервис

После установки в фоне работает `kaiten-watcher` — Docker-контейнер, который:

- Опрашивает заданные колонки Kaiten каждые N минут
- При появлении новой карточки отправляет уведомление в Пачку
- Если карточка с типом INCIDENT — помечает это в уведомлении
- Проверяет оформление карточки по шаблону (8 пунктов) через Claude AI
- Если оформление не прошло — пишет комментарий в карточку с перечнем нарушений
- После прохождения проверки оценивает карточку по 8 критериям (0–3 балла каждый)
- Записывает баллы в кастомные поля Kaiten
- Отправляет итоговое сообщение с оценками в Пачку

## Требования

- macOS (Intel или Apple Silicon)
- [Docker Desktop](https://www.docker.com/products/docker-desktop/) — для локальной установки
- Или VPS с Docker и SSH-доступом — для серверной установки

## Установка

Готового DMG нет — установка производится из исходников (см. раздел «Сборка из исходников»). После сборки:

### Шаг 1. Открыть

Дважды кликнуть по `Kaiten-Installer.dmg`, перетащить приложение в папку Applications.

> **Gatekeeper:** при первом запуске macOS может заблокировать приложение.  
> Решение: правой кнопкой на приложении → «Открыть» → «Открыть».

### Шаг 2. Пройти Wizard

Wizard проведёт через 9 шагов:

| Шаг | Содержание |
|-----|-----------|
| 1 | Приветствие. Если сервис уже установлен — сообщит об этом |
| 2 | Выбор места развёртывания: локально или VPS |
| 3 | Настройки Kaiten: токен, URL, Space ID, Column IDs |
| 4 | Настройки Пачки: токен, Chat ID |
| 5 | Anthropic API Key |
| 6 | Дополнительные настройки (интервал опроса, Google Sheet, ID полей Kaiten) |
| 7 | Проверка подключений к API |
| 8 | Развёртывание с выводом лога |
| 9 | Готово |

## Обязательные настройки

### Kaiten
| Параметр | Описание |
|----------|----------|
| API Token | Токен из настроек профиля Kaiten |
| Base URL | `https://api.kaiten.ru/api/latest` |
| Space ID | ID пространства (из URL пространства) |
| Column IDs | ID колонок для мониторинга через запятую |

### Пачка
| Параметр | Описание |

### Anthropic
| Параметр | Описание |
|----------|----------|
| API Key | Ключ из [console.anthropic.com](https://console.anthropic.com/) |

## Где хранятся данные

**Локальная установка:** `~/.kaiten-watcher/`

```
~/.kaiten-watcher/
├── config.env          # все настройки
├── docker-compose.yml
├── watcher.py
├── checker.py
├── state.json          # состояние: какие карточки уже обработаны
└── .installed          # маркер установки
```

**VPS:** `/opt/kaiten-watcher/` (или путь, указанный в Wizard)

## Изменение настроек

После редактирования `~/.kaiten-watcher/config.env` перезапустить контейнер:

```bash
cd ~/.kaiten-watcher && docker compose restart
```

## Переустановка

Удалить каталог с данными и запустить Wizard заново:

```bash
# Остановить контейнер
docker compose -f ~/.kaiten-watcher/docker-compose.yml down

# Удалить каталог установки
rm -rf ~/.kaiten-watcher/

# Запустить Kaiten Installer снова
```

## Сборка из исходников (для разработчиков)

```bash
# Клонировать репозиторий
git clone https://github.com/yurysaharov-cpu/kaiten-installer.git
cd kaiten-installer

# Установить зависимости (нужен Python 3.11+ и Homebrew)
brew install create-dmg
pip3 install -r requirements.txt

# Собрать DMG
bash build.sh
```

На выходе — `Kaiten-Installer.dmg` в корне проекта.

## Известные особенности

### Подколонки Kaiten не отслеживаются

Kaiten API возвращает карточки только из указанной колонки, не из вложенных подколонок. Если в Wizard указан ID родительской колонки, карточки в её подколонках будут пропущены.

Решение: указывать ID каждой подколонки через запятую в поле `Column IDs`. ID подколонки виден в URL при клике на неё в интерфейсе Kaiten.

### Пачка API — статус 403 на шаге проверки

Валидация обращается к `/chats/{chat_id}`. Если у токена нет scope `chats:read` — придёт 403, но это не ошибка: сервис использует только `messages:create`, для которого токен достаточен. Wizard считает проверку пройденной при любом статусе кроме 401.

### Docker Keychain на macOS

При пересборке контейнера из терминала без GUI-сессии Docker не может получить credentials из Keychain и выдаёт:

```
error getting credentials - err: exit status 1, out: `keychain cannot be accessed`
```

Решение — разблокировать Keychain перед командой:

```bash
security -v unlock-keychain ~/Library/Keychains/login.keychain-db
```

### Повторная установка после сброса

Если контейнер пересобирается с нуля (удалён `state.json`), при первом запуске сервис инициализирует состояние по текущим карточкам и не отправляет уведомления. Уведомления начнут приходить только по новым карточкам, появившимся после старта.
